## 数据库优化

### 1. 索引优化

确保关键查询字段都有适当的索引：

```sql
-- 资产表索引
CREATE INDEX idx_assets_ip_port ON assets(ip, port);
CREATE INDEX idx_assets_last_seen ON assets(last_seen);

-- ICP检测记录索引
CREATE INDEX idx_icp_checks_asset_id ON icp_checks(asset_id);
CREATE INDEX idx_icp_checks_check_time ON icp_checks(check_time);
CREATE INDEX idx_icp_checks_is_registered ON icp_checks(is_registered);
CREATE INDEX idx_icp_checks_composite ON icp_checks(is_registered, check_time);

-- 扫描任务索引
CREATE INDEX idx_scan_tasks_policy_id ON scan_tasks(policy_id);
CREATE INDEX idx_scan_tasks_status ON scan_tasks(status);
CREATE INDEX idx_scan_tasks_created_at ON scan_tasks(created_at);
```

### 2. 连接池配置

在配置文件中调整 MySQL 连接池参数：

```yaml
database:
  max_idle_conns: 10      # 最大空闲连接数
  max_open_conns: 100     # 最大打开连接数
  conn_max_lifetime: 3600 # 连接最大生命周期（秒）
```

### 3. 查询优化

- 使用批量插入而不是单条插入
- 避免 N+1 查询问题
- 使用 LIMIT 限制返回结果集大小
- 定期执行 `OPTIMIZE TABLE` 清理碎片

## Redis 优化

### 1. 内存配置

在 `configs/redis.conf` 中优化内存设置：

```conf
# 最大内存限制
maxmemory 2gb

# 内存淘汰策略 - 使用 LRU
maxmemory-policy allkeys-lru

# 持久化配置
save 900 1
save 300 10
save 60 10000

# AOF 配置
appendonly yes
appendfsync everysec
```

### 2. Stream 配置

优化 Redis Stream 的消费者组设置：

```go
// 设置合理的批次大小
streamReadOptions := &redis.XReadGroupArgs{
    Group:    "icp-workers",
    Consumer: workerID,
    Streams:  []string{streamKey, ">"},
    Count:    10,           // 每次读取10条
    Block:    5 * time.Second,
}
```

### 3. 连接池优化

```yaml
redis:
  pool_size: 100           # 连接池大小
  min_idle_conns: 10       # 最小空闲连接
  max_retries: 3           # 最大重试次数
  pool_timeout: 30         # 连接池超时（秒）
```

## HTTP 客户端优化

### 1. 连接复用

系统已启用 HTTP Keep-Alive 和连接池：

```go
// 在 pkg/utils/http.go 中
Transport: &http.Transport{
    MaxIdleConns:        100,  // 全局最大空闲连接
    MaxIdleConnsPerHost: 10,   // 每个主机最大空闲连接
    IdleConnTimeout:     90 * time.Second,
    DisableKeepAlives:   false, // 启用 Keep-Alive
}
```

### 2. 超时设置

根据实际情况调整超时参数：

```yaml
icp:
  probe_timeout: 10      # HTTP 探测超时（秒）
  screenshot_timeout: 30 # 截图超时（秒）
  
engines:
  fofa:
    timeout: 30          # FOFA API 超时
  hunter:
    timeout: 30          # Hunter API 超时
```

### 3. 重试策略

实现指数退避重试：

```go
maxRetries := 3
baseDelay := 1 * time.Second

for i := 0; i < maxRetries; i++ {
    resp, err := client.Do(req)
    if err == nil {
        return resp, nil
    }
    
    if i < maxRetries-1 {
        delay := baseDelay * time.Duration(1<<uint(i))
        time.Sleep(delay)
    }
}
```

## 并发处理优化

### 1. Worker 数量调整

根据 CPU 核心数和负载调整 Worker 数量：

```yaml
# docker-compose.yml
services:
  icp-worker:
    deploy:
      replicas: 4  # 根据 CPU 核心数调整
```

建议：
- CPU 核心数 ≤ 4: 1-2 个 Worker
- CPU 核心数 4-8: 2-4 个 Worker  
- CPU 核心数 > 8: 4-8 个 Worker

### 2. 协程池大小

在 Worker 中控制并发数：

```go
// 并发探测
concurrency := 50  // 根据网络带宽和目标响应能力调整
semaphore := make(chan struct{}, concurrency)

for _, task := range tasks {
    semaphore <- struct{}{}
    go func(t Task) {
        defer func() { <-semaphore }()
        process(t)
    }(task)
}
```

### 3. 批处理

批量处理任务以减少开销：

```go
batchSize := 100
for i := 0; i < len(items); i += batchSize {
    end := i + batchSize
    if end > len(items) {
        end = len(items)
    }
    batch := items[i:end]
    processBatch(batch)
}
```

使用 Prometheus 监控关键指标：


## 常见性能问题

### 问题1: 队列堆积

**症状**: Redis Stream 队列长度持续增长

**解决方案**:
- 增加 Worker 数量
- 提高 Worker 的并发处理能力
- 优化探测超时时间
- 检查网络连接问题

### 问题2: 数据库慢查询

**症状**: 数据库 CPU 使用率高，查询响应慢

**解决方案**:
- 添加缺失的索引
- 优化复杂查询
- 使用 EXPLAIN 分析查询计划
- 考虑读写分离

### 问题3: 内存占用过高

**症状**: OOM 错误，容器重启

**解决方案**:
- 减少批处理大小
- 限制并发数
- 及时释放不用的资源
- 使用 pprof 定位内存泄漏

### 问题4: HTTP 请求超时

**症状**: 大量超时错误

**解决方案**:
- 增加超时时间
- 实现请求重试
- 使用断路器模式
- 检查目标网络可达性

## 最佳实践总结

1. **合理配置资源**: 根据实际负载调整数据库、Redis 和应用的资源配置
2. **启用连接池**: 复用数据库和 HTTP 连接，避免频繁创建销毁
3. **批量操作**: 尽可能使用批量操作减少网络往返
4. **异步处理**: 使用队列和 Worker 模式处理耗时任务
5. **监控告警**: 建立完善的监控体系，及时发现性能瓶颈
6. **定期维护**: 定期清理历史数据，优化数据库表
7. **压力测试**: 上线前进行充分的性能测试
