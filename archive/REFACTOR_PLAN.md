# UniMap 代码重构计划

## 目标

本计划旨在通过代码重构提高 UniMap 项目的可维护性、性能和可靠性，主要针对三个核心领域：

1. **Web-only 适配器重构**：减少代码重复，提高可扩展性
2. **错误处理统一**：建立统一的错误处理机制，提高错误信息质量
3. **缓存优化**：实现动态缓存策略和 LRU 淘汰机制，提高性能

## 1. Web-only 适配器重构

### 现状分析

当前每个搜索引擎都有独立的 Web-only 适配器实现，导致代码重复率高：

- `FofaAdapterWebOnly`
- `HunterAdapterWebOnly`
- `ZoomEyeAdapterWebOnly`
- `QuakeAdapterWebOnly`
- `ShodanAdapterWebOnly`

每个适配器都有相似的结构和功能，只是针对不同引擎的特定实现有所不同。

### 重构方案

1. **创建通用 Web-only 适配器基类**：
   - `WebOnlyAdapterBase` 作为所有 Web-only 适配器的基类
   - 实现通用的 Web-only 模式逻辑
   - 提供可扩展的接口供具体引擎实现

2. **重构具体引擎适配器**：
   - 每个引擎适配器继承自基类
   - 只实现引擎特定的逻辑
   - 移除重复代码

3. **统一注册机制**：
   - 简化引擎注册逻辑
   - 自动处理 Web-only 模式的判断

### 预期成果

- 代码重复率降低 50% 以上
- 新引擎添加时只需实现核心逻辑
- 维护成本显著降低

## 2. 错误处理统一

### 现状分析

当前错误处理分散在各个模块，错误信息不够统一和详细：

- 错误格式不一致
- 错误分类不清晰
- 错误信息不够详细，难以定位问题
- 缺乏统一的错误处理策略

### 重构方案

1. **创建统一错误类型**：
   - 定义 `UnimapError` 结构体
   - 包含错误类型、错误码、错误信息、堆栈信息等字段
   - 实现标准错误接口

2. **错误分类**：
   - 网络错误
   - API 错误
   - 配置错误
   - 运行时错误
   - 业务逻辑错误

3. **错误处理工具**：
   - 错误包装函数
   - 错误转换函数
   - 错误日志工具

4. **统一错误响应**：
   - Web API 统一错误响应格式
   - 错误码映射
   - 友好的错误提示

### 预期成果

- 错误信息更加清晰和详细
- 错误定位更加准确
- 错误处理逻辑更加一致
- 用户体验得到改善

## 3. 缓存优化

### 现状分析

当前缓存机制存在以下问题：

- 缓存时间固定为 30 分钟，未根据不同场景调整
- 缺乏缓存大小限制，可能导致内存占用过高
- 缓存淘汰策略简单，未实现 LRU 等高效算法
- 不同类型的缓存（查询结果、截图等）使用相同策略

### 重构方案

1. **动态缓存策略**：
   - 根据查询类型、引擎类型自动调整缓存时间
   - 热门查询缓存时间较长，冷门查询缓存时间较短
   - 实现缓存时间的动态调整机制

2. **LRU 缓存实现**：
   - 实现基于 LRU 算法的缓存淘汰机制
   - 支持缓存大小限制
   - 自动淘汰最不常用的缓存项

3. **多级缓存**：
   - 内存缓存（热点数据）
   - 磁盘缓存（冷数据）
   - 缓存层次结构优化

4. **缓存监控**：
   - 缓存命中率统计
   - 缓存大小监控
   - 缓存性能分析

### 预期成果

- 缓存命中率提高 20% 以上
- 内存使用更加合理
- 系统响应速度更快
- 缓存管理更加智能化

## 实施步骤

### 第一阶段：Web-only 适配器重构（3-5 天）

1. 创建 `WebOnlyAdapterBase` 基类
2. 重构各个引擎的 Web-only 适配器
3. 更新引擎注册逻辑
4. 测试验证

### 第二阶段：错误处理统一（3-4 天）

1. 定义统一错误类型和分类
2. 实现错误处理工具函数
3. 重构现有代码中的错误处理
4. 测试验证

### 第三阶段：缓存优化（4-6 天）

1. 实现 LRU 缓存算法
2. 实现动态缓存策略
3. 集成到现有系统
4. 性能测试和调优

## 测试计划

1. **单元测试**：确保重构后的代码功能正常
2. **集成测试**：确保各模块协同工作正常
3. **性能测试**：验证缓存优化效果
4. **回归测试**：确保现有功能不受影响

## 风险评估

1. **代码兼容性**：重构可能影响现有功能
   - 缓解措施：充分的测试和代码审查

2. **性能影响**：缓存优化可能带来性能波动
   - 缓解措施：渐进式实施和监控

3. **维护成本**：重构后的代码需要适应期
   - 缓解措施：详细的文档和代码注释

## 预期收益

1. **代码质量**：代码更加简洁、可维护
2. **性能提升**：缓存优化带来响应速度提升
3. **开发效率**：减少重复代码，提高开发效率
4. **用户体验**：错误信息更加友好，系统更加稳定
