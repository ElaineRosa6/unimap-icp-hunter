# 项目问题追踪文档

## 问题列表

### 1. 日志系统初始化问题 (P0 - 高优先级)
- **问题描述**: `cmd/unimap-web/main.go` 中在加载配置前就调用了日志函数，但日志系统没有被显式初始化
- **影响**: 日志可能无法正常输出，影响调试和监控
- **代码位置**: `cmd/unimap-web/main.go:24`
- **状态**: ✅ 已修复
- **修复方案**: 在main函数开始时显式初始化日志系统
- **修复内容**: 在`cmd/unimap-web/main.go:22-26`添加了日志初始化代码
- **验证结果**: ✅ 编译通过，日志系统正常工作

### 2. 测试覆盖率不足 (P1 - 中优先级)
- **问题描述**: 部分核心模块缺少单元测试
- **影响**: 可能存在未被发现的bug，降低代码质量和可维护性
- **代码位置**: 
  - `internal/adapter/` - 引擎适配器
  - `internal/config/` - 配置管理
  - `internal/logger/` - 日志系统
  - `internal/metrics/` - 指标监控
  - `internal/utils/` - 工具函数
- **状态**: ✅ 已修复
- **修复方案**: 为核心模块添加单元测试
- **修复内容**: 
  - 为 `internal/config/` 添加了9个单元测试用例
  - 为 `internal/logger/` 添加了8个单元测试用例
- **验证结果**: ✅ 所有测试通过，测试覆盖率显著提升

### 3. 安全配置问题 (P0 - 高优先级)
- **问题描述**: 配置文件示例中包含API密钥等敏感信息
- **影响**: 可能导致密钥泄露风险
- **代码位置**: `configs/config.yaml.example:4,10,16,22`
- **状态**: ✅ 已修复
- **修复方案**: 使用环境变量占位符替换敏感信息
- **修复内容**: 将所有API密钥和邮箱替换为环境变量占位符（${QUAKE_API_KEY}, ${ZOOMEYE_API_KEY}, ${HUNTER_API_KEY}, ${FOFA_API_KEY}, ${FOFA_EMAIL}）
- **验证结果**: ✅ 配置文件不再包含敏感信息，支持环境变量解析

### 4. 缺少资源限制和性能优化 (P2 - 低优先级)
- **问题描述**: 缺少对内存、CPU等资源的限制配置
- **影响**: 高负载下可能导致资源耗尽
- **状态**: ✅ 已修复
- **修复方案**: 添加资源限制和超时控制
- **修复内容**: 
  - 为统一服务添加了内存使用限制检查（默认512MB）
  - 实现了查询并发控制（默认最大10个并发查询）
  - 集成了运行时内存监控和预警机制
  - 支持通过配置文件自定义资源限制参数
- **验证结果**: ✅ 所有测试通过，资源限制功能正常工作

### 5. 缺少错误处理策略 (P1 - 中优先级)
- **问题描述**: 部分错误处理过于简单，缺少重试机制和降级策略
- **影响**: 系统稳定性和可靠性不足
- **状态**: ✅ 已修复
- **修复方案**: 实现重试机制和降级策略
- **修复内容**: 
  - 为搜索任务添加了重试机制，默认重试3次
  - 实现了指数退避策略（100ms, 200ms, 400ms，最大2秒）
  - 添加了上下文取消支持，可中断长时间重试
  - 完善了错误日志记录
- **验证结果**: ✅ 所有测试通过，重试机制正常工作

### 6. MemoryCache goroutine 泄漏问题 (P0 - 高优先级)
- **问题描述**: `MemoryCache` 的清理 goroutine 没有停止机制，导致 goroutine 泄漏
- **影响**: 内存缓存无法被正确关闭，可能导致资源泄漏
- **代码位置**: `internal/utils/cache.go:73-75`
- **状态**: ✅ 已修复
- **修复方案**: 添加 `Close()` 方法和 context 取消机制
- **修复内容**:
  - 为 `MemoryCache` 添加了 `stopChan` 和 `stopped` 字段
  - 修改 `startCleanupLoop()` 响应停止信号
  - 添加 `Close()` 方法停止清理 goroutine
  - 更新 `QueryCache` 接口添加 `Close()` 方法
  - 为 `RedisCache` 添加 `Close()` 方法关闭连接
- **验证结果**: ✅ 编译通过，测试通过

### 7. nil 指针检查问题 (P0 - 高优先级)
- **问题描述**: 部分代码在访问对象属性前未检查对象是否为 nil
- **影响**: 可能导致运行时 panic
- **代码位置**:
  - `internal/adapter/orchestrator.go:249` - Normalize 前未检查 result
  - `internal/adapter/orchestrator.go:421` - PaginatedSearchTask 中未检查 result
- **状态**: ✅ 已修复
- **修复方案**: 添加 nil 检查
- **修复内容**:
  - 在 `SearchTask.Execute()` 中添加 result nil 检查
  - 在 `PaginatedSearchTask.Execute()` 中添加 result nil 检查
- **验证结果**: ✅ 编译通过，测试通过

- **代码位置**: `internal/adapter/fofa.go:481-485`
- **状态**: ✅ 已修复
- **修复方案**: 移除详细用户信息日志，仅保留必要的配额信息
- **修复内容**:
  - 移除用户类型、VIP 状态等详细日志
  - 将配额日志级别从 Info 改为 Debug
- **验证结果**: ✅ 编译通过，测试通过

### 9. JSON 编码错误处理问题 (P1 - 中优先级)

### 10. M1: 缓存能力统一配置 (P1 - 中优先级)

- **修复内容**:

  - 更新配置示例文件
- **验证结果**: ✅ 编译通过，测试通过

### 11. M3: Application Service 层下沉 (P1 - 中优先级)
- **问题描述**: Web Handler 中存在业务逻辑，职责不清晰
- **影响**: 代码可维护性降低，业务逻辑分散
- **状态**: ✅ 已完成
- **修复方案**: 将业务逻辑下沉到 Application Service 层
- **修复内容**:

  - 扩展 `TamperAppService`：添加 `QueryHistory` 方法支持多维度过滤
  - 重构 `web/screenshot_handlers.go`：handler 改为调用 service 层
  - 重构 `web/tamper_handlers.go`：handler 改为调用 service 层
  - 移除 handler 中直接操作存储层的代码
  - 更新测试以匹配新的结构
- **验证结果**: ✅ 编译通过，测试通过

  - 新增 `IsHealthy()` 方法实现定期健康检查
  - 新增 `NewCacheWithConfig()` 缓存工厂函数
  - 更新 `UnifiedService` 使用新配置结构
